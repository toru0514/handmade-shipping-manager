# 宅急便コンパクト「スマホで送る」自動化 — 実現可能性 調査メモ

> 調査日: 2026-06-11 / ステータス: **検討のみ（実装前）**
> 結論: **iOSシミュレータは不要。PC上のモバイルエミュレーションで実現可能性が高い。決済直前まで自動化し、決済はユーザーが行う方針とする。**

---

## 1. 背景・目的

ヤマト運輸の宅急便コンパクトでは、PUDOステーション等での発送に「**スマホで送る**」（2次元コード提示）が必要だが、これが**スマホ限定**のため、現状の自動化は**氏名・住所の登録までで止めている**。

iOSシミュレータを使って「スマホで送る」を自動で実現できないかを検討する。

---

## 2. 現状（なぜ止まっているか）

現実装 `src/infrastructure/external/playwright/YamatoPudoPage.ts` は、実は**「スマホで送る」のフローには到達していない**。

- アクセス先は `member.kms.kuronekoyamato.co.jp`（クロネコメンバーズ **会員サイト**）の **アドレス帳登録**。
- `#NEXT_BTN` クリック＝**アドレス帳登録完了**をもって「成功」とし、QR/送り状番号は**フォールバック値**（1x1透明PNG / `ADDRESS-BOOK-REGISTERED`）を返している（`YamatoPudoPage.ts:103-107`）。
- 「スマホで送る」の発送・QR発行は **別サイト** `https://sp-send.kuronekoyamato.co.jp/smpTaqWeb/...` で行われる。

→ **アドレス帳登録と発送フローは別物**。現コードのセレクタ・遷移は発送フローには流用できない。これが「止まっている」本質。

---

## 3. 「スマホで送る」の正体（調査結果）

| 項目 | 事実 |
|---|---|
| 形態 | **専用アプリではなくモバイルWebサービス**（`sp-send.kuronekoyamato.co.jp`） |
| 推奨環境 | iPhone: iOS 12以上 / **Safari**、Android: 8以上 / Chrome |
| PCで使えない理由 | デバイス判定でブロックし「スマホでQRをスキャンして」と表示されるだけ |
| **PC回避策（実証済み）** | Chrome **DevToolsのモバイルデバイスエミュレーション**を有効化すると、ログイン → 情報入力 → 支払い → **QR発行までPCで完結**できる。**user-agent変更だけでは不十分**で、touch / viewport を含むデバイスエミュレーションが必要 |

### 重要な含意
DevToolsのモバイルエミュレーションで足りる＝**Playwrightの `devices['iPhone 13']` 相当（UA + viewport + isMobile + hasTouch + deviceScaleFactor 一括設定）で同じことができる**。
→ **iOSシミュレータの本物のiOS環境は今回オーバースペック。**

---

## 4. iOSシミュレータの評価 → 不要

| 観点 | 評価 |
|---|---|
| 現環境 | **未インストール**（Command Line Toolsのみ、`simctl` 不可）。フルXcode（数GB〜十数GB）が必要 |
| App Storeアプリ | シミュレータは**App Storeアプリを実行不可**。ただし今回はWebサービスなので**そもそも不要** |
| 自動化 | Safari操作の自動化に **Appium + WebDriverAgent** 等の重いセットアップが必要 |
| 実行環境 | **macOS GUI必須。CI / Vercel / サーバーレスで実行不可**（本番の自動発送パイプラインに乗らない） |
| 唯一の強み | 「本物のiOS環境」で厳格なデバイス判定も突破。だが**DevToolsエミュレーションで突破できる実例がある＝不要** |

→ **iOSシミュレータはコストが高く、本番パイプラインに乗らず、得られる強みが今回は不要。採用しない。**

---

## 5. Browserless でのブロック問題（既知の制約）

- 過去に **Browserless 経由で自動操作した際はブロックされた**実績がある。
- 原因の候補:
  - **データセンターIP**（Browserless / クラウド）からのアクセスを弾いている可能性。
  - ヘッドレス / 自動化ブラウザのフィンガープリント検知。
  - モバイルエミュレーション未設定でPC判定されていた可能性。
- 含意:
  - 「スマホで送る」フローは **ローカル（自宅回線等）の Chromium + モバイルエミュレーション** で動かす前提で検討するのが現実的。
  - そもそも本フローはGUIブラウザを伴うため、**Vercel/サーバーレスの本番パイプラインには直接乗らない**。**ローカル実行 or 専用環境**を想定する。

---

## 6. 決済の方針

- 「スマホで送る」は送料を**オンライン決済（クレジットカード / Apple Pay 等）**する。
- 自動化は **決済画面の手前まで** とし、**決済自体はユーザーが手動で行う**。
  - → 実課金を自動化に含めないため、**安全・規約面のリスクを大きく低減**できる。
  - Apple Pay はSafari前提でエミュレーションでは通らない可能性があるが、**決済をユーザーに委ねるため問題にならない**。

---

## 7. 推奨アプローチ（コストの低い順）

1. **【第一候補】Playwright モバイルエミュレーション**
   - 既存の `ChromiumBrowserFactory` に `devices['iPhone 13']` 相当のcontextオプション（userAgent / viewport / isMobile / hasTouch）を渡す。
   - `sp-send.kuronekoyamato.co.jp` フロー用の**新しいページオブジェクト**を実装。
   - **決済画面手前まで**自動操作 → ユーザーに引き渡し。
   - **ローカルChromium**で実行（Browserless はブロック実績があるため避ける）。
2. **【確実だが手動寄り】実機iPhone** — 最も確実だが自動化が困難。
3. **【最終手段】iOSシミュレータ + Appium** — 1, 2が技術的に塞がれた場合のみ。

---

## 8. リスク・未解決事項

- **`sp-send` 側フローの未調査**: 画面遷移・セレクタは未調査。現コードの会員サイト用セレクタは流用不可で、新規調査・実装が必要。
- **デバイス判定の強度**: DevToolsエミュレーションで通る実例はあるが、ヤマト側の判定強化で塞がれる可能性は残る。
- **ローカル実行前提**: Browserlessブロックの件から、本番自動化基盤（サーバーレス）との折り合いを別途検討する必要。
- **規約面**: 自動化・スクレイピングの是非。

---

## 9. 次のステップ（実装前のPoC）

実課金を伴わない範囲で、以下を順に確認する（実装は別途）。

1. **ローカル Playwright + iPhoneエミュレーション**で `sp-send.kuronekoyamato.co.jp` のエントリ〜ログインが「PCブロックされず」モバイルフローで開けるかを確認（**決済手前まで**）。
2. フロー全体の画面遷移・セレクタを洗い出し、ページオブジェクト設計を起こす。
3. 「決済画面手前で停止 → ユーザーへ引き渡し」のUX/連携方法を設計。
