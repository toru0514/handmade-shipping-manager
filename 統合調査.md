# 統合調査: handmade-shipping-manager + pfauto-app

調査日: 2026-03-25
更新日: 2026-03-25（最終判断追記）

---

## 最終判断

### 結論: 統合は実行可能。HSMをベースに進めてよい。

両アプリ（HSM側・PFA側）の調査結果を突合した結果、**技術的な阻害要因はゼロ**。以下の全条件が揃っている。

| 判断項目 | 結果 |
|---------|------|
| Next.jsバージョン一致 | 両方 16.1.6 |
| DB競合 | なし（同一Supabase、テーブル名重複なし） |
| URL競合 | なし（`/dashboard/*` vs `/orders/*` etc.） |
| 環境変数競合 | `GOOGLE_SHEETS_SPREADSHEET_ID` のみ → リネームで解決 |
| UIライブラリ統一 | MUI統一は容易（PFA側はshadcn/ui未使用、カスタムTailwind 8〜14ファイル） |
| Playwright排他 | 同時実行不可として扱う → ジョブキューで直列化 |
| アーキテクチャ整合 | PFA側をヘキサゴナルに変換してからHSMに統合 |

### PFA側調査との差異・補足

PFA側の調査で追加で判明した有用な情報：

1. **middleware統合が必要**（PFA側レポートで指摘）
   - Next.jsの `middleware.ts` はプロジェクトに1つしか置けない
   - 両方にミドルウェアがあるため、マージが必要
   - 認証をSupabase Authに統一すれば、PFA側のmiddlewareベースで統合可能

2. **Google Sheetsクライアントの実装差異**が明確化
   - PFA: `googleapis` SDK使用（レート制限・キャッシュ付き）
   - HSM: REST API直接呼び出し + JWT自前実装
   - → PFA側のSDKベース（レート制限付き）に統一が望ましい

3. **Vercelとローカルの役割分離**
   - Playwright必要なエンドポイントを明確に分離する設計が重要
   - Vercel: 閲覧系（ダッシュボード、売上分析、注文一覧）
   - ローカル: 実行系（出品下書き、注文取得、ラベル発行）

### 統合の進め方（確定版）

**HSMをベースアプリとし、PFAの機能を移植する方式で進める。**

ただし、移植前にPFA側で先にやるべき準備がある：

```
Step 1: PFA側の準備（PFAリポジトリで実施）
  ├── UIをMUIに変換（8〜14ファイル）
  └── ディレクトリ構造をヘキサゴナルに変換

Step 2: HSM側の準備（HSMリポジトリで実施）
  ├── パッケージマネージャをpnpm/Bunに移行
  ├── 環境変数名のリネーム（GOOGLE_SHEETS_SPREADSHEET_ID → SHIPPING_SPREADSHEET_ID）
  └── 認証をSupabase Authに統一

Step 3: 統合（HSMリポジトリで実施）
  ├── PFAのdomain/application/infrastructure層のコードを移植
  ├── PFAのページ・APIルートを移植
  ├── DIコンテナ拡張
  ├── middleware統合
  ├── Google Sheetsクライアント統一
  └── ジョブキュー統合（Playwright排他制御）

Step 4: 検証
  ├── 全機能の動作確認
  ├── テスト統合・実行
  └── Vercel閲覧 + ローカル実行の動作確認
```

### リスクまとめ（最終）

| リスク | 重大度 | 対策 | 状態 |
|--------|--------|------|------|
| UIライブラリ混在 | ~~高~~ → 解消 | MUI統一（PFA側のUI少量） | 方針決定済み |
| 認証統合 | 中 | Supabase Authに統一 | 要実施 |
| middleware統合 | 中 | PFA側ベースでマージ | 要実施 |
| Sheetsクライアント重複 | 低〜中 | googleapis SDK版に統一 | 要実施 |
| Playwright排他 | ~~中~~ → 解消 | ジョブキューで直列化 | 方針決定済み |
| パッケージマネージャ | 低 | pnpm/Bunに統一 | 要実施 |
| Reactバージョン差 | 低 | 最新版に統一 | 自動解決 |

**不可能な要因: なし。統合を進めてよい。**

## 2. 共通点

- **同一のNext.jsバージョン** (16.1.6) — 統合の最大の利点

- **Tailwind CSS** — 両方で使用
- **Vitest** — テストフレームワーク共通
- **Slack Webhook** — 通知に使用
- **TypeScript strict mode** — 両方で有効

## 3. MUI統一の実現性

### 結論: 難しくない（工数: 小〜中）

PFAには**shadcn/uiのプリビルトコンポーネントが実際にはインストールされていない**。UIは全てTailwindベースのカスタムコンポーネント（8〜10個）で構成されている。

### PFAのUIコンポーネント一覧と対応するMUIコンポーネント

| PFAコンポーネント | 内容 | MUI対応 |
|------------------|------|---------|
| `dashboard-content.tsx` (33KB) | メインダッシュボード・商品テーブル | `DataGrid` or `Table` + `Card` |
| `add-product-modal.tsx` | 商品追加モーダル | `Dialog` + `TextField` |
| `sidebar.tsx` | ナビゲーション | `Drawer` or `Box` |
| `mobile-header.tsx` | モバイルヘッダー | `AppBar` |
| `product-edit-form.tsx` (13KB) | 商品編集フォーム | `TextField` + `Select` |
| `copy-product-dialog.tsx` | コピー確認ダイアログ | `Dialog` |
| `image-picker-dialog.tsx` | 画像選択 | `Dialog` + `ImageList` |
| `wood-form-modal.tsx` | 材料フォーム | `Dialog` + `TextField` |
| `woods-content.tsx` | 材料一覧 | `Grid` + `Card` |
| `wood-detail.tsx` | 材料詳細 | `Card` + `Typography` |

### 変換のポイント

- Tailwindクラス → MUIの `sx` prop に変換（機械的作業）
- カスタムモーダル（fixed overlay実装） → MUI `Dialog`（自動でoverlay処理）
- `lucide-react` アイコン → `@mui/icons-material` に置換
- CSS変数テーマ → MUI `createTheme()` に統合

### PFA側から削除可能な依存

```
- class-variance-authority  (shadcn/ui用、不要に)
- clsx                      (MUIでは不要)
- tailwind-merge             (MUI移行後は不要)
- lucide-react               (@mui/icons-materialに置換)
```

## 6. 懸念点・リスク

### 6.1 認証システムの統合（重大度: 高）

| 問題 | 詳細 |
|------|------|
| **異なる認証方式** | HSM: `APP_PASSWORD`ベースのシンプル認証 / PFA: Supabase Auth + JWT |
| **ミドルウェア** | PFAにはNextAuth + middleware.tsによるルート保護がある |

**対策案**:
- PFAのSupabase Auth方式に統一するのが望ましい（HSMも既にSupabase接続あり）
- 統合初期はPFA側の認証ロジックで両方のルートを保護

### 6.2 アーキテクチャの変換（重大度: 中）

| 問題 | 詳細 |
|------|------|
| **レイヤー構成の違い** | PFAは `adapters/` `application/` `components/` のフラット構成 → ヘキサゴナルに変換 |
| **Server Actions** | PFAは `actions.ts` でServer Actionsを多用。HSMはAPI Routeベース |

**対策案**:
- PFAのコードをHSMのレイヤー構成にリファクタリングして配置
- Server Actionsは初期段階ではそのまま維持しても動作する

### 6.3 Google Sheetsクライアントの重複（重大度: 低）

| 問題 | 詳細 |
|------|------|
| **2つのSheetsClient** | 両アプリにGoogle Sheets APIクライアントがある。PFA側にはレート制限やキャッシュ機能がある |

**対策案**:
- 初期段階は2つ共存（環境変数でスプレッドシートID分離）
- 将来的にPFA側のレート制限付きクライアントをベースに統合

### 6.4 Playwrightの排他制御（重大度: 低 → 方針決定済み）

**同時実行はしない（不可能として扱う）。**
- ジョブキュー（PFA側に既存実装あり）で排他制御
- UIレベルで同時実行を防ぐガード追加

### 6.5 Sentryの追加（重大度: 低）

PFAはSentry統合がある（client, server, edge設定）。HSMにはない。
- 統合後にSentryを全体に適用するのは有益だが、設定ファイル(`sentry.*.config.ts`, `instrumentation.ts`)の追加が必要

### 6.6 パッケージマネージャ移行（重大度: 低）

HSMのnpmからpnpm/Bunへの移行が必要。
- `package-lock.json` → `pnpm-lock.yaml` or `bun.lockb`
- CIやスクリプトの`npm`コマンドを置換
- 移行自体は機械的で低リスク

## 7. 統合不可能な要因

**なし。** 統合を不可能にする技術的要因は発見されなかった。

- 同一のNext.jsバージョン（16.1.6）
- 同一のDB（Supabase）
- 機能的な重複がない（発送管理 vs 出品管理）
- ルート（URL）の衝突がない
- UIライブラリ統一はコンポーネント数が少なく現実的
- Playwright同時実行は不可として排他制御

## 8. 統合の推奨ステップ

### Phase 1: 基盤準備
1. パッケージマネージャをpnpm（またはBun）に統一
2. PFAの依存パッケージをHSMに追加（Sentry等）
3. 環境変数名のリネーム（スプレッドシートID関連）
4. 認証方式の統一（Supabase Auth推奨）

### Phase 2: PFA側ヘキサゴナル化 + コード移植
1. PFAのGoogle Sheetsアダプタ → `infrastructure/adapters/persistence/` に配置
2. PFAのuse cases → `application/usecases/` に配置
3. PFAのPlaywrightページオブジェクト → `infrastructure/external/playwright/` に配置
4. PFAのジョブキュー・ユーティリティ → `infrastructure/lib/` に配置
5. DIコンテナの拡張

### Phase 3: UI移植（MUI変換）
1. PFAの8〜10個のコンポーネントをMUIに変換
2. `presentation/components/products/`, `woods/`, `jobs/` として配置
3. ナビゲーションに出品管理メニューを追加
4. ページルート追加（`/products`, `/woods`, `/jobs`）

### Phase 4: 統合テスト・仕上げ
1. 両機能の動作確認
2. Playwright排他制御の確認
3. 認証フローの統合テスト
4. E2Eテストの統合

## 9. 結論

**統合は技術的に可能であり、阻害要因はない。**

MUI統一もPFA側のUIコンポーネントが少数（8〜10個）かつshadcn/uiを実際に使っていないため、Tailwind→MUI変換のみで対応可能。

PFA側を先にヘキサゴナルアーキテクチャ化し、ディレクトリ構造を合わせてからHSMに統合するのが最もスムーズ。
