# 統合計画: handmade-shipping-manager (HSM) + pfauto-app (PFA)

作成日: 2026-03-25
更新日: 2026-03-25（PFA側レビュー反映）

---

## 前提

- **ベースアプリ**: HSM（handmade-shipping-manager）
- **移植元**: PFA（pfauto-app）
- **最終成果物**: 1つのNext.jsアプリに統合
- **認証**: 両方ともSupabase Auth済み（HSM: `supabase.auth.signInWithPassword` + middleware、PFA: Supabase Auth + JWT）
- **DB**: 同一Supabase、テーブル競合なし
- **UIライブラリ**: MUIに統一
- **パッケージマネージャ**: Bun
- **Playwright**: 同時実行不可（ジョブキューで直列化）
- **Vercel**: 閲覧専用（Playwright処理はローカル）

> **認証に関する補足**: HSMは過去に `APP_PASSWORD` + セッション方式だったが、2026-03-23のコミット (85c07cb) で Supabase Auth に移行済み。現在のコードに `APP_PASSWORD` / `APP_SESSION_SECRET` の参照は存在しない。

---

## フェーズ概要

```
Phase A (並行作業)          Phase B (統合)           Phase C (検証)
┌─────────────────┐
│ PFA側: MUI変換   │
│ （構造変換は不要）│──┐
└─────────────────┘  │    ┌──────────────┐    ┌──────────────┐
                     ├───→│ コード統合     │───→│ 検証・仕上げ   │
┌─────────────────┐  │    └──────────────┘    └──────────────┘
│ HSM側: 基盤準備   │──┘
└─────────────────┘
```

- **Phase A**: 各アプリで独立して作業（並行可能）
- **Phase B**: HSMリポジトリにPFAコードを統合（ヘキサゴナル構造への配置もここで行う）
- **Phase C**: 統合後の検証・調整

```

/_next/*

```

#### Google Sheetsクライアント

初期段階は**2つのクライアントを共存**させる:
- HSM側: 既存の `SheetsClient.ts`（REST直接、`SHIPPING_SPREADSHEET_ID` を使用）
- PFA側: 移植する `ProductSheetsClient.ts`（googleapis SDK、`PRODUCT_SPREADSHEET_ID` を使用）

将来的にgoogleapis SDK版に統一（Phase Bのスコープ外）。

PFA側のサイドバー (`sidebar.tsx`) とモバイルヘッダー (`mobile-header.tsx`) はHSM既存のGlobalNavに統合し、独立コンポーネントとしては不要。

## PFA側レビューで追加された補足事項

> 以下はPFA側コードレビューで検出された、計画本文に反映済みの追加事項。HSM側で実装時に参照。

### 1. 出品プラットフォーム環境変数（BASE/iichi分）

PFAはCreema/minne以外にBASE/iichiにも対応している。A-2-2のプレースホルダーと統合後の環境変数セクションにBASE/iichiの変数を追加済み。

### 2. カテゴリマッピング（`lib/categories/`）

PFAの `lib/categories/` ディレクトリにはMinne/Creemaのカテゴリ別名→IDマッピングデータがある。出品下書き作成（CreemaDraftPage等）で必要。B-3で `src/infrastructure/lib/categories/` に配置する。

### 3. scriptsディレクトリ

PFAのスクリプトのうち以下をHSMに移植する（B-3に追記済み）:
- `scripts/seed-admin.ts` — Supabase Auth初回ユーザー作成。HSMでも共用可能。
- `scripts/test-sheets.ts` — 商品スプシ接続テスト。`test-product-sheets.ts` にリネームして配置。

### 4. Playwrightファイルの分類

PFAのPlaywright関連ファイルは2種類に分かれる（B-3に詳細追記済み）:
- **アプリケーションコード**（page objects: ジョブキューから呼ばれる出品処理） → `src/infrastructure/external/playwright/` に配置
- **E2Eテスト**（spec files, shared utils） → `playwright/tests/` に配置

HSM側の既存構造（`MinnePage.ts`, `CreemaPage.ts` 等がアプリコードとして `src/infrastructure/external/playwright/` にある）と同じ方針。

### 5. session-provider.tsx

PFAの `components/providers/session-provider.tsx` は現在パススルー（中身なし）のため、**移植しない**。HSM既存のSupabase Auth基盤で代替。B-5に明記済み。

### 6. MicroCMS画像管理

PFAの `/dashboard/images/upload` ページはMicroCMS連携で画像管理を行っている。B-1の依存パッケージ、環境変数セクションに追加済み。

### 7. PFA商品スプシのシート名環境変数

PFAのGoogle Sheetsは複数タブ構成（共通, Creema, minne, BASE, iichi）。`PRODUCT_SPREADSHEET_ID` だけでなく、`PRODUCT_WORKSHEET_TITLE`（メインタブ名）と `PRODUCT_COMMON_SHEET`（共通タブ名）も必要。環境変数セクションに追加済み。

### 8. `app/dashboard/layout.tsx` の再構成

PFAの `layout.tsx` には独自のサイドバー/モバイルヘッダーが組み込まれている。HSMのGlobalNavに統合する方針のため、そのまま移植ではなくHSMのレイアウト体系に合わせて書き直す必要がある。B-6に注記済み。

### 9. Sentry統合時の追加ファイル（任意）

Sentryを全体適用する場合、PFAから以下のファイルも移植が必要:
- `instrumentation.ts` — サーバー起動時のSentry初期化
- `sentry.client.config.ts` — クライアント側Sentry設定
- `sentry.server.config.ts` — サーバー側Sentry設定
- `sentry.edge.config.ts` — Edge Runtime用Sentry設定
- `next.config.ts` のSentry Webpack Plugin設定をHSMの `next.config.ts` にマージ
